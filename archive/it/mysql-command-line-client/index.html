Il client da riga di comando mysql

Informazioni sul client da riga di comando mysql

mysql è una semplice shell SQL (che fa uso di GNU readline). Supporta sia l'utilizzo interattivo sia quello non interattivo. Quando è usato in modo interattivo, i risultati delle query sono presentati come tabelle con caratteri ASCII. Quando è usato in modo non interattivo (per esempio come filtro), i risultati sono presentati con un carattere tab come separatore. Il formato dell'output può essere cambiato tramite le opzioni.

Se con i grandi insiemi di risultati la memoria non è sufficiente, si usi l'opzione --quick. Essa fa sì che mysql estragga i risultati dal server una riga alla volta, piuttosto che riceverli tutti insieme e inserirli in un buffer in memoria in attesa di mostrarli. Questo si può ottenere anche utilizzando la funzione dell'API C mysql_use_result() nella libreria client/server, invece di mysql_store_result().

Utilizzare mysql è molto semplice. Per invocarlo dal prompt dei comandi:

shell> mysql nome_db

Oppure:

shell> mysql --user=nome_utente --password=pwd nome_db

Poi si possono scrivere le istruzioni SQL, che devono terminare con “;”, \g, o \G, e infine premere Invio.

A partire da MySQL 5.1.10, se si preme Control-C mysql tenta di terminare l'esecuzione dell'ultima istruzione. Se questo non è possibile, o se Control-C viene premuto di nuovo prima che il comando venga terminato, mysql termina l'esecuzione. Precedentemente, Control-C terminata mysql in ogni caso.

E' possibile eseguire le istruzioni SQL in uno script (file batch) in questo modo:

shell> mysql nome_db < script.sql > output.tab

mysql Options

Le opzioni possono essere specificate dalla riga di comando o nel file my.cnf, nei gruppi [mysql] e [client]. L'elenco completo delle opzioni supportate si può visualizzare così:

shell> mysql --verbose --help

L'output che si ottiene potrebbe essere leggermente diverso, a seconda della versione in uso e delle impostazioni nel file my.cnf.

Il file mysql_history

Su Unix, il client mysql registra ogni istruzione eseguita in un file di cronologia. Per default, questo si chiama .mysql_history e si trova nella directory home dell'utente. Per utilizzare un altro file, lo si specifichi nella variabile di ambiente MYSQL_HISTFILE.

Il file .mysql_history dovrebbe essere protetto in modo restrittivo, perché potrebbero esservi scritte informazioni sensibili, come il testo di alcune istruzioni SQL che contengono delle password. Si veda la sezione 5.3.2.2, “End-User Guidelines for Password Security”.

Se non si desidera mantenere un file di cronologia, prima si rimuova .mysql_history se esiste, poi si usi una delle seguenti tecniche:

  • Si imposti la variabile MYSQL_HISTFILE a /dev/null. Perché questa impostazione abbia effetto ad ogni login, la si inserisca in un file eseguito all'avvio della shell.
  • Si crei .mysql_history come collegamento simbolico a /dev/null:
shell> ln -s /dev/null $HOME/.mysql_history

Questo deve essere fatto una sola volta.

Suggerimenti su mysql

Questa sezione spiega alcune tecniche che possono essere d'aiuto per utilizzare mysql in modo più efficiente.

Mostrare i risultati verticalmente

A volte i risultati delle query sono più leggibili se mostrati verticalmente, invece che in modo orizzontale come avviene di solito. Le query possono essere mostrate in verticale terminandole con il carattere \G invece che con un punto e virgola. Ad esempio, i valori di testo più lunghi che includono dei caratteri di nuova riga a volte sono molto più semplici da leggere se mostrati verticalmente:

mysql> SELECT * FROM mails WHERE LENGTH(txt) < 300 LIMIT 300,1\G
*************************** 1. row ***************************
  msg_nro: 3068
    date: 2000-03-01 23:29:50
time_zone: +0200
mail_from: Monty
    reply: monty@no.spam.com
  mail_to: "Thimble Smith" <tim@no.spam.com>
      sbj: UTF-8
      txt: >>>>> "Thimble" == Thimble Smith writes:
Thimble> Hi.  I think this is a good idea.  Is anyone familiar
Thimble> with UTF-8 or Unicode? Otherwise, I´ll put this on my
Thimble> TODO list and see what happens.
Yes, please do that.
Regards,

Usare l'opzione --safe-updates

I principianti potrebbero trovare utile l'opzione di avvio --safe-updates (o --i-am-a-dummy, che ha lo stesso effetto). Serve nei casi in cui si ha lanciato un'istruzione 'DELETE FROM nome_tabella' ma si ha dimenticato la clausola WHERE. Normalmente, questo comando eliminerebbe tutte le righe di una tabella. Con --safe-updates, è possibile eliminare le righe solo specificando i valori delle chiavi che li identificano. In questo modo, si prevengono alcuni incidenti.

Se si usa l'opzione --safe-updates, mysql esegue la seguente istruzione quando si connette al server MySQL:

SET sql_safe_updates=1, sql_select_limit=1000, sql_max_join_size=1000000;

L'istruzione SET ha i seguenti effetti:

  • Non è permesso eseguire istruzioni UPDATE o DELETE a meno che non si specifichino le chiavi nella clausola WHERE o si aggiunga una clausola LIMIT (o entrambe).Per esempio: 
UPDATE nome_tabella SET nome_colonna_non_chiave=val WHERE colonna_chiave=val;
UPDATE nome_tabella SET cone_colonna_non_chiave=val LIMIT 1;
  • Il server limita tutte le grandi SELECT che producono più di 1,000 righe, a meno che non includano una clausola LIMIT.
  • Il server abortisce le SELECT multi-tabella che probabilmente dovrebbero esaminare più di 1 milione di combinazioni.

Per specificare limiti diversi da 1,000 e 1,000,000, si possono utilizzare le opzioni --select_limit e --max_join_size:

shell> mysql --safe-updates --select_limit=500 --max_join_size=10000

Disabilitare la riconnessione automatica di mysql

Se il client mysql perde la connessione al server mentre si invia un'istruzione, immediatamente cerca di riconnettersi e inviare nuovamente il comando. Tuttavia, anche se mysql riesce a ristabilire la connessione, quella precedente è terminata e tutti gli oggetti di sessione sono persi: tabelle temporanee, modalità di autocommit e variabili di sessione o definite dall'utente. Inoltre, tutte le transazioni interrotte vengono annullate. Questo comportamento può essere pericoloso. Si consideri l'esempio seguente, dove il server viene arrestato e riavviato tra la prima e la seconda istruzione, senza che l'utente lo sappia:

mysql> SET @a=1;
Query OK, 0 rows affected (0.05 sec)
mysql> INSERT INTO t VALUES(@a);
ERROR 2006: MySQL server has gone away
No connection. Trying to reconnect...
Connection id:    1
Current database: test
Query OK, 1 row affected (1.30 sec)
mysql> SELECT * FROM t;
+------+
| a    |
+------+
| NULL |
+------+
1 row in set (0.05 sec)

La variabile utente @a si perde con la connessione, e dopo la riconnessione è indefinita. Se è importante che mysql termini con un errore quando la connessione viene interrotta, si avvi il client con l'opzione --skip-reconnect.
