Criteri di rilascio

This is a translation of the outdated version of the Release Criteria. For the current release criteria see the English variant.

Il team di MariaDB può solo promettere di coprire i bug di Aria o del codice che ha sviluppato per MariaDB. Per i bug di MySQL®, non può fornire alcuna garanzia. Tuttavia, per quei bug di MySQL che affliggono direttamente Aria o sono correlati con Aria, il team di MariaDB molto probabilmente sviluppa un fix o un workaround.

Le politiche di rilascio di MariaDB seguono i MySQL release criteria. Inoltre sono state adottate le seguenti regole per gli stadi di maturità:

Le politiche di rilascio per tutte le versioni dello Storage Engine Aria e di tutti i rilasci di MariaDB (anche Alpha):

  • Tutti i rilasci di MariaDB dovrebbero essere privi di bug che possano causare problemi in operazioni che sono 'normali' per la maggior parte degli utenti.
  • I rilasci di MariaDB comprendono tutte le modifiche nell'albero principale di MySQL fino alla build stabile più recente.
  • Se un rilascio contiene bug critici (per esempio, se c'è un bug molto brutto che vogliamo correggere una volta per tutte e far uscire un fix, mentre stiamo correggendo altri bug), questi saranno documentati nel file KNOWN_BUGS.txt.

Impegni per i rilasci Beta

  • Aria/MariaDB è feature-complete secondo quanto concordato dai captain di MariaDB per questo rilascio.
  • Nessun bug serio conosciuto, eccetto quelli classificati come feature request.
  • Nessun bug corretto dall'ultimo rilascio che ha avuto dei cambiamenti nel codice degni di nota.

Impegni per i rilasci Gamma/RC

  • Nessun bug serio conosciuto, eccetto quelli classificati come feature request.
  • Nessun bug corretto dall'ultimo rilascio che ha avuto dei cambiamenti nel codice degni di nota.
  • Crediamo che il codice sia pronto per un utilizzo diffuso (basandoci sulle segnalazioni dei bug), ma vogliamo fare altri test prima di dichiararlo stabile.

Impegni per i rilasci Stable/GA

  • Nessun bug serio conosciuto, eccetto quelli classificati come feature request.
  • Nessun bug corretto dall'ultimo rilascio che ha avuto dei cambiamenti nel codice degni di nota.
  • Crediamo che il codice sia pronto per un utilizzo diffuso (basandoci sulle segnalazioni dei bug).

Nei rari casi in cui un bug non può essere corretto in uno specifico rilascio (perché è un errore di progettazione o il fix potrebbe causare altri bug, magari peggiori), lo documenteremo nel file KNOWN_BUGS.txt che si trova nella distribuzione di MariaDB. Tuttavia, cercheremo di mantenere al minimo questo tipo di bug aperti.
