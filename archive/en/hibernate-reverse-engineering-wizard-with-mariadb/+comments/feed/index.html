The problem lies in the JDBC driver for MariaDB and the Hibernate Tools. If you crank up Hibernate's logging you'll see messages like this:</p> <p><code>2014-12-03 21:16:00,214 [main] DEBUG reveng.JDBCReader - Ignoring table locations of type BASE TABLE</code></p> <p>This is because the MariaDB JDBC driver was reporting the types of the table as BASE TABLE, however Hibernate Tools will only generate reverse engineered files for database objects of type TABLE or VIEW.</p> <p>From Hibernate Tools 4.3.1.CR1: org.hibernate.cfg.reveng.JDBCReader#processTables():</p> <pre> if ( ("TABLE".equalsIgnoreCase(tableType) || "VIEW".equalsIgnoreCase(tableType) /*|| "SYNONYM".equals(tableType) */) ) { //|| ... } else { log.debug("Ignoring table " + tableName + " of type " + tableType); } </pre> <p>To work around it I just used the legacy MySQL JDBC driver to do the reverse engineering and then switched to the MariaDB driver for actual usage.</p> John Gibsonhttps://mariadb.com/kb/en/hibernate-reverse-engineering-wizard-with-mariadb/+comments/1389Re: Hibernate Reverse Engineering wizard with MariaDBhttps://mariadb.com/kb/en/hibernate-reverse-engineering-wizard-with-mariadb/+comments/932<p>Just to update this. </p> <p>I gave up trying to make the reverse engineering work.</p> <p>There was already an ORM in place (Toplink).
