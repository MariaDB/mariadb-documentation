MariaDB Knowledge Base Comments for: Questions about backup database using idb fileshttps://mariadb.com/kb/en/questions-about-backup-database-using-idb-files/+comments/feed/en-usTue, 26 Dec 2023 00:39:26 +0000Re: Questions about backup database using idb fileshttps://mariadb.com/kb/en/questions-about-backup-database-using-idb-files/+comments/3986<p>The .ibd files never store table ID. Index pages do store index ID and transaction ID, and every page contains a tablespace ID and a log sequence number (LSN). All these will be rewritten by ALTER TABLE…IMPORT TABLESPACE.</p> <p>I would like to improve this:</p> <ul start="1"><li><a href="https://jira.mariadb.org/browse/MDEV-11658">MDEV-11658</a> Simpler, faster IMPORT of InnoDB tables </li><li><a href="https://jira.mariadb.org/browse/MDEV-15225">MDEV-15225</a> Can't import .ibd file with temporal type format differing from mysql56_temporal_format </li></ul> <p>The reason why arbitrarily copying files while InnoDB is running is not safe is that the InnoDB buffer pool basically acts as a write-back cache. Also, plain FLUSH TABLES or FLUSH TABLES…WITH READ LOCK has no effect on InnoDB, because InnoDB can perform some writes in the background, on files that have not been touched by SQL for quite a while. These background writes will be synchronized and suppressed by FLUSH TABLES…FOR EXPORT.</p> <p>MySQL 5.7 implemented CREATE TABLESPACE for InnoDB, but MariaDB did not follow with that. It is hard to design and implement all aspects of that in a consistent way. For example, you cannot export and import such shared tablespaces in MySQL.</p> Marko Mäkelähttps://mariadb.com/kb/en/questions-about-backup-database-using-idb-files/+comments/3986
