Although that is reasonable in OLTP environments, it is not acceptable in OLAP .</p> <p>I think my use case is very common. We have a 1500 columns table where each row stores all the data of a single transaction (IP addresses, messages sent/received, subscriber data,...). Many of the columns of the table are TINYTEXT. In order to create the table in the columnstore engine, we changed all those columns to varchar(256), but it wasn't possible because we kept getting the error:</p> <p>"Row size too large. The maximum row size for the used table type, not counting BLOBs, is 65535".</p> <p>The only way to create the table was to reduce the amount of columns to less than 350. We can't loose that many columns in production environments.</p> <p>That's the reason why I believe <a href="https://jira.mariadb.org/browse/MCOL-267">MCOL-267</a> and <a href="https://jira.mariadb.org/browse/MCOL-356">MCOL-356</a> should have a higher priority.</p> José Caviereshttps://mariadb.com/kb/en/columnstore-data-types/+comments/2671
