MariaDB Knowledge Base Comments for: TIMESTAMPhttps://mariadb.com/kb/en/timestamp/+comments/feed/en-usMon, 07 Aug 2023 18:51:17 +0000TIMESTAMP & timezone problemshttps://mariadb.com/kb/en/timestamp/+comments/3653<p>When using the TIMESTAMP type in timezone settings with daylight savings, you must be aware that:</p> <ul start="1"><li>Storing data in the column will be ambiguous at the end of daylight savings, because the local-time hour gets repeated twice. </li></ul> <ul start="1"><li>At the start of daylight savings, when clocks are turned forwards, there's a "missing hour". Technically times in this hour aren't valid, but if you do try to store a time from this missing hour in a TIMESTAMP column and then read it back, you WON'T get out what you put in! (Contrary to what the MySQL docs state). </li></ul> <ul start="1"><li>If you ORDER BY your timezone column, it will order by the internal UTC representation; <em>displayed</em> times can therefore be out of order when daylight savings ends. </li></ul> <ul start="1"><li>HOWEVER, any computation will use the derived local time, even in the ORDER BY clause, so if ts is a TIMESTAMP column, "ORDER BY ts" "ORDER BY &lt;column number&gt;", or "ORDER BY UNIX_TIMESTAMP(ts)" give UTC-time ordering, but "ORDER BY ts+0", "ORDER BY ts + INTERVAL 1 SECOND", "ORDER BY CAST(ts AS datetime)" will be ordered by <em>local</em> time, which may result in a different row ordering! </li></ul> <ul start="1"><li>Similarly, comparisons between timestamp values, i.e. <em>ts1</em>&lt;<em>ts2</em> etc are always carried out using local-time values. So the result of this comparison can CHANGE depending on the timezone you set, if the values are on different sides of the turn-the-clocks-back point: IT IS NOT A COMPARISON BETWEEN UTC TIMESTAMPS!!! To do that, compare UNIX_TIMESTAMP(<em>ts1</em>)&lt;UNIX_TIMESTAMP(<em>ts2</em>) </li></ul> <p>These problems can be largely obviated by setting the server timezone to UTC (temporarily, if you prefer), i.e.:</p> <pre class="fixed"> SET @oldTZ := @@time_zone; SET @@time_zone := '+00:00'; # # Do something with timestamps here; it'll work reliably[*] as everything is in UTC # SET @@time_zone := @oldTZ; </pre><p>[*] Actually, not entirely true, as there may also occasionally be leap-seconds, represented by hh:59:59 repeating twice. So events with these timestamps may be more than one second apart, and there isn't a 1-1 correspondence between actual seconds and displayed seconds; and when storing decimal seconds (milliseconds, etc), once again all the local-times-are-out-of order issues can arise (even in the UTC timezone). See the following page in the MySQL docs: <a href="https://dev.mysql.com/doc/refman/8.0/en/time-zone-leap-seconds.html">https://dev.mysql.com/doc/refman/8.0/en/time-zone-leap-seconds.html</a></p> Dean Trowerhttps://mariadb.com/kb/en/timestamp/+comments/3653Timezoneshttps://mariadb.com/kb/en/timestamp/+comments/2433<p>It's important to note that MariaDB (and MySQL) both have the notion of a server timezone - when writing an explicit value to a timestamp column, time data is converted from the server timezone into UTC for storage, then converted back to server time when reading.</p> <p>If you're planning on using the timestamp data type to hold timestamps which already in UTC, it's probably worth setting your server timezone to UTC. Alternatively, pay the size penalty and use the DATETIME type, which isn't affected by timezones.</p> <p><a href="https://mariadb.com/kb/en/mariadb/time-zones/">https://mariadb.com/kb/en/mariadb/time-zones/</a></p> Andy Walkerhttps://mariadb.com/kb/en/timestamp/+comments/2433Re: TIMESTAMP end range is incorrect.https://mariadb.com/kb/en/timestamp/+comments/1715<p>Thanks, this has been fixed (the limit is actually 2038-01-19 05:14:07)</p> Ian Gilfillanhttps://mariadb.com/kb/en/timestamp/+comments/1715Re: TIMESTAMPhttps://mariadb.com/kb/en/timestamp/+comments/1256<p>thanks to update the documentation.</p> Max Dixhttps://mariadb.com/kb/en/timestamp/+comments/1256Re: TIMESTAMPhttps://mariadb.com/kb/en/timestamp/+comments/1255<p>I missed that this has been possible since MySQL 5.6.5 and <a href="/kb/en/mariadb-1001-release-notes/">MariaDB 10.0.1</a>, so it is already possible in the current stable MariaDB release.
