The default delimiter is semicolon. So, in your initial example, the client sends everything from <code>CREATE</code> and up and including <code>SET NEW.ClGender ='U'</code>, which of course does not make a complete valid query. This is what the server complains about in the error message -- it finds emptiness where it expects <em>something else</em>. </p> <p>When you change the delimiter and use it at the end of the trigger definition, the client understands that the whole definition should be sent as a whole, and the server is able to interpret it correctly. </p> <p>It is perfectly fine to use non-default delimiters when you define stored procedures (including triggers); you don't need to invent tricks to avoid it. It is not always possible anyway.</p> <p>But if you are curious, in your case you could phrase it like this:</p> <pre> CREATE TRIGGER trig_client_gndr BEFORE INSERT ON WM_CLIENT FOR EACH ROW SET NEW.ClGender = IF(NEW.ClGender NOT IN ('M','F'),'U',NEW.ClGender); </pre> <p>I think it should have worked.</p> Elena Stepanovahttps://mariadb.com/kb/en/trigger-keeps-creating-error/+comments/1289Re: Trigger keeps creating errorhttps://mariadb.com/kb/en/trigger-keeps-creating-error/+comments/1288<p>Thank you,</p> <p>That worked perfectly.
