So to exploit the password you need to run the command exploit from within the container and if you've exposed that access, there's not much protection left anyway.</p> <p>Combined with MYSQL_RANDOM_ROOT_PASSWORD=1 when starting the container is one way of ensuring the password isn't exposed in the inspect, however it is in the logs, so maybe not much better.</p> <p>Alternate to mysql_secure_install is just `set password =` for the root user.</p> <p>If you do get stuck without a root password, using the [init_file|<a href="https://mariadb.com/kb/en/server-system-variables/#init_file">https://mariadb.com/kb/en/server-system-variables/#init_file</a>] option with the right create/alter user/grant in the file it points to to get access back.</p> Daniel Blackhttps://mariadb.com/kb/en/root-password-change-how-can-i-stop-the-mysqld-in-the-mariadb-container/+comments/5159Re: Root Password Change - How can I stop the mysqld in the mariadb containerhttps://mariadb.com/kb/en/root-password-change-how-can-i-stop-the-mysqld-in-the-mariadb-container/+comments/5158<p>I ended up setting no password, and then using mysql_secure_installation to set the password.</p> James Stewart Millerhttps://mariadb.com/kb/en/root-password-change-how-can-i-stop-the-mysqld-in-the-mariadb-container/+comments/5158
