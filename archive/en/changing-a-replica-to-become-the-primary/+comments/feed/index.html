MariaDB Knowledge Base Comments for: Changing a Replica to Become the Primaryhttps://mariadb.com/kb/en/changing-a-replica-to-become-the-primary/+comments/feed/en-usMon, 05 Aug 2024 03:35:22 +0000Re: Changing a Slave to Become the Masterhttps://mariadb.com/kb/en/changing-a-replica-to-become-the-primary/+comments/5056<p>Is `FLUSH TABLES WITH READ LOCK;` the best way to lock the primary from receiving new queries? I had replications fails to restart after switchover many times. Then, I started using Orchestrator and never had an issue again. Turns out, Orchestrator [1] only set `SET GLOBAL read_only = ON;` on the primary before switchover. Then I read this answer <a href="https://dba.stackexchange.com/questions/74670/revoke-write-privileges-vs-setting-database-to-readonly">https://dba.stackexchange.com/questions/74670/revoke-write-privileges-vs-setting-database-to-readonly</a> which explain the advantages of read_only over FTWRL.</p> <p>Other thing, in last chapter, "Now you can upgrade the new master to a newer version" should be "Now you can upgrade the <strong>OLD</strong> master to a newer version"</p> <p>Thanks for your great documentation, Laurent Indermühle</p> <p>[1] <a href="https://github.com/openark/orchestrator/blob/master/docs/topology-recovery.md#graceful-master-promotion">https://github.com/openark/orchestrator/blob/master/docs/topology-recovery.md#graceful-master-promotion</a></p> Laurent Indermühlehttps://mariadb.com/kb/en/changing-a-replica-to-become-the-primary/+comments/5056
