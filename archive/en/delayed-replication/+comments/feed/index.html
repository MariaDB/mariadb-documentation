Seconds behind masterhttps://mariadb.com/kb/en/delayed-replication/+comments/3589<p>One more field of interest for monitoring is Seconds_Behind_Master. When initially the slave is in sync the four fields would say the following:</p> <pre class="fixed">MariaDB [test]&gt; show slave status\G Seconds_Behind_Master: 0 SQL_Delay: 30 SQL_Remaining_Delay: NULL Slave_SQL_Running_State: Slave has read all relay log; waiting for the slave I/O thread to update it </pre><p>Here 30 seconds is my preference for MASTER_DELAY.</p> <p>Once the master has updated some data </p> <pre class="fixed">MariaDB-master [test]&gt; insert into t set a=1; Query OK, 1 row affected (0.01 sec) </pre><p>and it is replicated the fields show is progressed maintaining</p> <pre class="fixed">Seconds_Behind_Master + SQL_Remaining_Delay == SQL_Delay </pre><p>property:</p> <pre class="fixed">MariaDB-slave [test]&gt; show slave status\G Seconds_Behind_Master: 1 SQL_Delay: 30 SQL_Remaining_Delay: 29 Slave_SQL_Running_State: Waiting until MASTER_DELAY seconds after master executed event ... MariaDB-slave [test]&gt; show slave status\G Seconds_Behind_Master: 28 SQL_Delay: 30 SQL_Remaining_Delay: 2 Slave_SQL_Running_State: Waiting until MASTER_DELAY seconds after master executed event 1 row in set (0.01 sec) MariaDB-slave [test]&gt; show slave status\G Seconds_Behind_Master: 0 SQL_Delay: 30 SQL_Remaining_Delay: NULL Slave_SQL_Running_State: Slave has read all relay log; waiting for the slave I/O thread to update it </pre>Andrei Elkinhttps://mariadb.com/kb/en/delayed-replication/+comments/3589
